# Brick Breaker Android — Product Brief

## Product
A polished Android Brick Breaker game built as a complete, shippable mobile game rather than a prototype.

## Primary Goal
Finish the existing `brick-breaker-android` project into a fun, stable, visually coherent, fully playable Brick Breaker game with a clear progression loop and production-ready structure.

## Core Experience
The player controls a paddle at the bottom of the screen and keeps the ball in play to break bricks, clear levels, survive mistakes, and progress through increasingly interesting stages. The game should feel responsive, readable, satisfying, and mobile-friendly.

## Definition Of Done
The app is considered complete when it includes:
- a working and polished core brick-breaker gameplay loop
- reliable paddle, ball, brick, collision, scoring, life, win, and loss systems
- multiple levels or waves with increasing challenge
- clear start, restart, pause, win, and game-over flows
- stable Android builds that run correctly on target devices
- sound effects and basic visual polish
- a sensible architecture that is maintainable
- no major gameplay-breaking bugs
- no placeholder-level roughness in the main user experience

## Quality Bar
This is not just a coding exercise or tech demo. The target is a complete, enjoyable, polished game with good feel and production-minded implementation decisions.

## Constraints
- Use the existing repository as the source of truth for current implementation status.
- Prefer incremental, testable, reviewable changes.
- Preserve working features unless there is a strong reason to replace them.
- Avoid unnecessary rewrites when targeted improvements will suffice.
- Android-first experience.
- Human involvement should stay minimal except where true human testing is needed.

## Autonomy Expectations
The orchestrator/planner should evaluate the existing repo, determine current status from actual code, refine the roadmap as needed, and then drive the project toward completion step by step. Codex is the implementation worker, not the product strategist.

## Areas To Evaluate
- current gameplay implementation status
- project architecture and code organization
- rendering/game loop approach
- input handling
- physics and collision reliability
- level system
- score/lives/progression system
- UI and menus
- sound/music
- save/high-score capability if appropriate
- test/build/run reliability
- missing polish and content

## Success Criteria
A user should be able to install the game, understand it quickly, enjoy playing it, and experience it as a real finished Brick Breaker game rather than an unfinished prototype.
